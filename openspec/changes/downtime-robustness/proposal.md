## Why

A span that is never closed is currently counted as work **right up to the moment someone looks at it**. `pairSpans` closes an orphan open span at `checkTime`, and every caller passes `Date.now()` (`tenant-do.ts:316,397`; `getWeek` default at `:244`). Because `computeDay` then clamps that span into each day of the week, a machine that goes down at 16:00 on Friday without emitting `idle` does not merely fill Friday to midnight — it fills **Saturday and Sunday completely, and Monday up to the current time**. Open the week view on Sunday evening and the weekend reads as ~56 hours of work.

The daemon already emits `heartbeat` events every 5 minutes precisely so that "unclosed spans can be bounded to the last heartbeat" (the existing `activity-daemon` requirement *Heartbeats bound crash damage*). **The backend never uses them for that.** `heartbeat` is in the `PRESENCE` set (`worktime.ts:63-68`), so it can *open* a span and is otherwise a no-op; nothing closes a span at the last heartbeat. The evidence needed to bound the damage is already in the database, unread.

The daemon's own recovery is real but incomplete. `StateMachine::recover` closes a stale span back-dated to the last heartbeat — but it runs **only at process start** (`main.rs:211`). Sleep and hibernate do not restart the process: the poll thread is frozen mid-`sleep()` and resumes hours later, and since `step()` compares nothing against wall-clock elapsed time, a user who touches the keyboard on wake produces a fresh sample, stays `Active`, and **the entire suspend is billed as work**. That is the reported symptom. The daemon also reads time only from `SystemTime` (`main.rs:24`), so it cannot distinguish a suspend from an NTP correction or a manual clock change.

## What Changes

- **The daemon becomes the authority on when a span ended.** It has ground truth the backend structurally lacks — whether it was suspended, when it resumed, when the last real input was — and its `idle` is exact rather than bounded. Explicit daemon events always win over anything the backend infers.
- **The backend bounds an open span only as a provisional floor**, at the last liveness evidence plus a grace allowance, computed at read time and never written back. It exists for the one case the daemon cannot cover — a machine that never returns — and is superseded as soon as buffered events arrive. It deliberately errs toward under-counting, because the backend cannot distinguish "machine is dead" from "machine is working behind a broken network" and must not fabricate hours in either case.
- **Closing a span dirties every day it previously covered**, not just the day of the arriving event. Today `ingest` marks only `localDayStart(e.ts)` dirty (`tenant-do.ts:152`), so a late `idle` for Friday 16:00 fixes Friday and leaves the phantom Saturday and Sunday rollups sealed and wrong. This is load-bearing rather than cosmetic: once the backend's bound is provisional, recompute-on-arrival *is* the correctness story for a network outage.
- **An unclosed span is presented as provisional, not as confirmed time.** Heartbeats say the user was active *at least* until the last one; they say nothing about after. A period ended by the bound is marked as such, shows when the machine was last seen, and re-adapts as later evidence arrives — becoming an ordinary measured period once the daemon's own closing event lands. Presenting an inferred end identically to an observed one would assert a certainty the data does not carry.
- **Idle is back-dated exactly, from the reported idle duration.** Back-dating already works (`state_machine.rs:114-116`), but anchors to the last poll that saw fresh input, quantising the result to the poll interval. The operating system already reports the exact answer — at a tick with idle time `X`, inactivity began at `now − X` — so the idle-timeout path uses that instead, while the lock path keeps the last-input anchor.
- **Reconciliation anchors to the most recent evidence, not the coarsest.** `recover()` currently prefers `last_heartbeat` over `last_active_time` (`state_machine.rs:91`), but `last_active_time` is written every tick (15 s) while heartbeats are 5 minutes apart — so the daemon back-dates less accurately than its own local state allows. Anchoring to the later of the two cuts the residual over-count on abrupt power loss from ~5 minutes to ~15 seconds, using state that is written locally and is therefore immune to network conditions.
- **The daemon detects suspend/resume in-process** by comparing elapsed monotonic time against elapsed wall-clock time each tick. A gap far exceeding the poll interval is downtime: the daemon closes the span back-dated to its last heartbeat and re-opens at resume, applying the same reconciliation `recover()` does at startup.
- **The daemon reads a monotonic clock alongside the wall clock**, so suspends, NTP corrections, and manual clock changes are distinguishable rather than conflated — including backwards jumps, which must never produce a negative-length span.
- **Best-effort OS power and shutdown handlers** emit a clean `idle` before the machine goes down (Windows `WM_POWERBROADCAST` / console-close; Linux logind `PrepareForSleep` and `SIGTERM`), flushing the outbox first. These are an optimisation on top of the gap detection, **never the sole mechanism** — a lost or unfired signal must degrade to the heartbeat bound, not to a phantom span.
- **`minInactivitySec` becomes a backend constant too; `minActivitySec` stays configurable.** The inactivity threshold is the second cost lever — a short value means more confirmed transitions and so more ingested events — and it doubles as this design's boundary between downtime that is absorbed and downtime that is reconciled, which must not vary per account. Fixing it is what makes the activity threshold safe to leave settable: with idle confirmation pinned, transitions cannot occur more often than once per interval, so write volume is capped regardless. `minActivitySec` is also the one of the three that is a real preference — how much sustained input counts as returning to work — and it gains the validation it currently lacks.
- **`heartbeatSec` stops being an account setting and becomes a backend constant.** Heartbeat frequency drives ingest write volume linearly: at the 5-minute default an account uses roughly 1% of the Durable Objects daily row-write allowance, but a 1-second value would generate ~350,000 row writes/day and breach the free tier — and the field is currently unvalidated. It is not a preference anyone holds, so rather than validating a floor it stops being settable at all, while `/config` keeps serving it so the daemon has a single source of truth. A value that cannot be set cannot be misconfigured.
- **Local and transmitted liveness are separated explicitly.** Local evidence is persisted every poll (15 s) purely to reconstruct the end timestamp after an ungraceful shutdown; the wire heartbeat stays coarse at 5 minutes because the daemon — not the heartbeat — decides where spans begin and end. This is what makes the ~1% write footprint affordable at no cost in accuracy.
- **Outbox hardening**: writes become atomic (temp + rename) so a crash mid-write cannot corrupt the queue into an unstartable daemon; the queue is bounded and chunked so a long offline period cannot grow one request past the ingest limit and wedge permanently.
- Existing behaviour that is deliberately kept: back-dating, hysteresis debouncing, and the durable outbox with monotonic `batch_seq` — all of which already work.

### Modified Capabilities
- `worktime-calculation`: an open span is bounded by its machine's last liveness evidence plus a grace allowance, instead of running to the time of observation — provisionally, always superseded by an explicit close.
- `activity-daemon`: downtime is detected in-process via monotonic-vs-wall-clock divergence, not only at startup; power/shutdown signals emit a clean close as a best-effort fast path; the outbox is crash-safe and bounded.
- `tenant-storage`: a change to a span's extent marks every day that span covered for recomputation, so corrections cannot leave stale sealed rollups behind; and daemon protocol timing ceases to be account-configurable, so the free tier cannot be breached by configuration.
- `web-ui`: a period whose end was inferred rather than observed is shown as provisional with a last-seen time, and re-adapts as later evidence arrives.

## Impact

- **Calc** (`backend/src/worktime/worktime.ts`): `pairSpans` gains per-machine liveness tracking and a bound for the open-span close; `heartbeat` keeps its `PRESENCE` role for opening but additionally advances a "last seen alive" marker.
- **Storage** (`backend/src/tenant-do.ts`): `ingest` dirty-marking widens from the event's own day to the span range affected; sealing and the 120-day prune window are unchanged.
- **Daemon** (`daemon/crates/flexitracker-daemon/`): `state_machine.rs` gains a `Tick` carrying monotonic elapsed time plus gap reconciliation shared with `recover()`; `main.rs` gains the monotonic clock source, signal/power handlers, and graceful-shutdown flush; `outbox.rs` gains atomic persist, a size bound, and batch chunking.
- **Settings**: the existing `heartbeatSec` becomes load-bearing for the backend bound (it is already fetched by the daemon and stored per account), so the grace allowance derives from it rather than from a new constant.
- **Tests**: daemon unit tests for suspend/resume, clock jumps in both directions, and outbox crash-safety; backend unit tests for heartbeat-bounded closure and multi-day dirty marking; an e2e scenario covering a machine that disappears mid-span.
- No wire-schema change — `heartbeat` already exists in `flexitracker-core`, `backend/src/schema.ts`, and `docs/wire-schema.md`.
